[1] 保密
本文档中包含的所有信息，都必须绝对保密。收到本RFP的所有厂商不得以任何方式向第三方传播、散布或授权其它方使用本文档的任何信息，同时上海芯源创新中心需对服务提供商应答的方案建议书和财务报价单保密并禁止传播。
关于保密义务及相关约束事项，均以双方签署生效的《上海芯源创新中心 CIM项目 保密协议（一般保密义务）- 赛美特 - signed》文件约定为准。
[1] SOW
为配合上海芯源创新中心CIM系统（EAP、FDC、RMS、RCM、APC、MES、SPC、AMS、PMS、RTD、AMA、YMS、DMS、RPT、FMS）的战略部署，中心亟需完善CIM内容与能力建设，已解决当前的如下问题：
项目选型困难：软硬件厂商种类繁多，15个CIM子系统运维管理独立，面临松散、繁多的技术栈；各家厂商的解决方案各异、独立，资源联合交付的标准化程度不匹配
无法引入先进技术：现中心软件聚焦于OA、ERP等技术领域，并未从半导体Fab视角部署CIM系统，难以满足未来生产需求并全栈对制造业信息技术进行孵化；在使用和管理上均难以互通，造成基础设施和能力平台孤岛
难以迅速响应业务变化：基础设施扩容周期漫长，主要提供系统软件部署、增量功能二次开发、IaaS资源（虚拟机、存储、商用负载均衡、CDN等），中间件、数据库和大数据等应用从场景需要额外立项采购软件和硬件进行部署交付
无法支撑信创工作目标达成：服务于国家战略，需要导入信创解决方案以完成国家信创工作的部署
可运维/监控/扩展：兼容中心现有的运维监控体系，同时具备较好的扩展自定义能力以满足运维和运营侧个性化定制需求。
CIM系统作为上海芯源创新中心与生产制造紧密联系的一部分，需有效解决当前无工业软件系统可用的状态，为工业软件、中间件、数据库、大数据和AI类应用的部署与支撑生产提供有效支撑，同时为后期开发测试和生产环境的搭建奠定基础。

[3] 典设的技术架构：
PlantU 平台为自研业务平台，采用 C# 桌面客户端、RavenCast 自有通信中间件、Java 后端服务和 Oracle 数据库组成整体技术架构。
平台通过 RavenCast 实现前后端解耦通信，支持同步调用和异步消息；后端 Java 服务可部署多个节点，并通过 RavenCast 的多活和负载均衡能力实现服务级高可用。
<TABLE>
| 逻辑层 | 技术实现 | 主要职责 |
| 表现层 | C# 桌面客户端 | 用户交互、业务操作、请求封装、结果展示 |
| 通信层 | RavenCast | 前后端通信、同步调用、异步消息、路由转发、负载均衡 |
| 服务层 | Java 服务 | 业务逻辑处理、流程控制、接口处理、数据访问 |
| 数据层 | Oracle Database | 业务数据、配置数据的存储与持久化 |
| 高可用层 | RavenCast 多活 + Java 多节点 | 服务多活、流量分发、横向扩展 |
</TABLE>

[3] 典设的技术架构：
PlantU 平台为自研业务平台，采用 C# 桌面客户端、RavenCast 自有通信中间件、Java 后端服务和 Oracle 数据库组成整体技术架构。
平台通过 RavenCast 实现前后端解耦通信，支持同步调用和异步消息；后端 Java 服务可部署多个节点，并通过 RavenCast 的多活和负载均衡能力实现服务级高可用。
<TABLE>
| 逻辑层 | 技术实现 | 主要职责 |
| 表现层 | C# 桌面客户端 | 用户交互、业务操作、请求封装、结果展示 |
| 通信层 | RavenCast | 前后端通信、同步调用、异步消息、路由转发、负载均衡 |
| 服务层 | Java 服务 | 业务逻辑处理、流程控制、接口处理、数据访问 |
| 数据层 | Oracle Database | 业务数据、配置数据的存储与持久化 |
| 高可用层 | RavenCast 多活 + Java 多节点 | 服务多活、流量分发、横向扩展 |
</TABLE>

[3] 典设的技术架构：
总体架构：
AMA 系统基于流程自动化平台建设，采用“前端静态资源 + Gateway 网关 + 后端 AP 应用服务 + 中间件 + 数据库 + 外部系统集成”的分层架构。用户通过浏览器访问 Nginx 代理的前端页面，接口请求统一进入 Gateway，再路由到 Process-Engine、调度服务、连接源服务、日志与监控等后端能力模块。
逻辑层划分：
系统逻辑上分为接入层、应用服务层、流程运行层、集成适配层和数据持久层。接入层负责 /wfa/ 静态资源代理、/engine/api/v1/ 接口入口和统一路由；应用服务层负责流程模型、Block/Action 配置、触发器、流程实例、任务、日志和权限管理；流程运行层负责 BPMN 解析、事件驱动执行、网关判断、变量上下文和异常处理；集成适配层负责数据库、HTTP、WebService、消息中间件、Redis、文件、远程命令和工业协议连接；数据持久层负责模型、实例、任务、日志、连接源和系统配置持久化。
核心模块：
AMA 依托系统内置流程平台、流程引擎、规则引擎、数据集成、调度服务、通知服务、运行日志和监控能力。流程平台提供模型设计、Block 编排、EventTrigger 触发和自定义页面能力；流程引擎负责实例启动、Action 调度、变量流转和流程状态推进；规则引擎负责网关条件、参数规则和分支决策；数据集成模块提供 DB、HTTP、Kafka、RocketMQ、Redis、MongoDB、SFTP/SSH、CORBA、TIBCO RV、MessageBus、RTD 等外部系统访问能力。
执行机制：
Process-Engine 启动流程实例后初始化运行上下文，发布 ExecuteActionEvent，由异步监听器基于 Java 21 虚拟线程执行节点任务。每个节点通过 IAction 策略模式按 java_class 反射加载具体 Action，执行结果写回流程变量，再由引擎根据 BPMN 连线或网关判断推进到下一节点。流程结束时触发锁释放、数据持久化、外部响应等 EndEvent 扩展。
接口与协议：系统对前端和第三方系统提供 HTTP/REST JSON 接口，内部服务通过 Gateway、OpenFeign 和 Nacos 服务发现完成路由与调用。外部系统集成协议覆盖 JDBC、HTTP/HTTPS、SOAP、Redis RESP、Kafka、RocketMQ、SFTP/SSH、CORBA/IIOP、TIBCO RV、MongoDB Wire Protocol 等；流程变量支持 ${变量名} 形式的动态映射，用于接口参数、消息体、SQL 参数和文件路径。
技术框架：后端基于 Java 21、Spring Boot 3.3.x、Spring Cloud 2023.x、Spring Cloud Alibaba、MyBatis-Plus、HikariCP、Redisson、Caffeine、Groovy、XXL-JOB、MapStruct、Fastjson2、SpringDoc OpenAPI 和 Actuator 构建。流程运行数据优先使用本地 Caffeine 缓存提升执行性能，流程结束后按配置持久化到系统数据库；Redis 主要用于 Spring Cache、分布式数据结构和锁相关能力。

总体部署：
生产环境建议采用高可用分层部署，按 DB、AP 和中间件三类服务器进行资源隔离。DB 使用专用数据库服务器；AP 使用专用应用服务器集群，部署 Gateway、Process-Engine、XXL-JOB 等服务；中间件使用共享中间件服务器集群，部署 Nacos、Redis、Nginx 等基础组件。各层之间通过内网访问，并通过防火墙或安全组限制端口范围。
DB 服务器：
DB 专用服务器保存流程定义、流程实例、任务记录、运行日志、连接源配置、用户角色、触发器配置和业务扩展配置。默认推荐 MySQL 8.1，可根据项目要求适配 PostgreSQL 或 Oracle。生产环境建议配置主从或高可用、定期备份、慢 SQL 分析、容量告警和数据库审计。
AP 应用服务器：
AP 服务器部署 Gateway、Process-Engine、XXL-JOB Executor/Admin 等应用。Gateway 作为统一 API 入口承接 /engine/api/v1/** 请求；Process-Engine 负责流程执行和 Block/Action 调度；XXL-JOB 负责定时流程、周期任务和任务调度。AP 服务建议至少 2 台以上部署，通过 Nacos 注册发现和负载均衡策略实现横向扩展。
中间件服务器：
中间件服务器部署 Nacos 集群、Redis 集群和 Nginx。Nacos 提供配置中心与注册中心，建议 3 节点集群；Redis 建议部署 3 主 3 从集群，用于缓存、分布式数据结构和锁能力；Nginx 用于前端静态资源代理，访问路径为 /wfa/**，并可配合 Keepalived 或外部负载均衡实现入口高可用。
网络与端口：
典型端口包括 Nginx 80/443、Gateway 8080 或项目规划端口、Process-Engine 8083、Nacos 8848/9848/9849、Redis 6379/6380 及 Cluster Bus 16379/16380、MySQL 3306、XXL-JOB Admin 8080、Actuator/监控端口按安全策略开放。生产环境建议只开放必要端口，AP 到 DB 和中间件均使用内网访问。

[3] 典设的技术架构：
数据层：设计人员创建报表数据源。
应用层：设计人员进行报表设计，管理人员配置用户和权限体系。
展示层：普通用户在前端执行报表的查询、分析、打印、 导出、填报等操作，支持 PC、平板 、移动端、大屏等设备，兼容主流浏览器。
FineReport 是纯 Java 软件，具有良好的跨平台兼容性，支持跟各类业务系统进行集成，支持各种操作系统，支持主流 Web 应用服务器。前台是纯 HTML 展现，无需安装任何插件。
FineReport 报表系统，从数据全链路流程来看，包括了「部署」、「数据准备」、「报表制作」、「报表使用」、「报表管理/ 集成」五大核心部分，FineReport 支持通过多种连接方式连接不同数据源，所有的报表制作工作都在设计器中完成，并最终通过服务器解析展现给用户。
FineReport 报表系统主要由「报表设计器」和「报表服务器」两部分构成：
报表设计器：可以进行表格、图形、参数、控件、填报、打印、导出等报表中各种功能的设计， 是集「报表应用开发」、「调试」、「部署」的一体化平台。
报表服务器：指用在 web 环境中解析报表的 Servlet 形式的服务器，用户通过浏览器和报表服务器进行应用交互。

故障严重等级定义：
“严重等级 1” : 意味着重大问题（整体系统停止）比如造成系统崩溃（用户返回到操作系统级），要求系统重新启动（持续和反复出现系统停机），或者由系统停止 造成系统运行上的缩减，由应用故障造成的用户数据被破坏了或者不可恢复，主要功能不可用并没有已知的解决方案（这里的解决方案是指解决或者绕过问题的方法）。
“严重等级2” : 意味着高优先度问题，严重损害了系统的主要功能，但并不妨碍业务运行和关键工程。 这包括系统丢失冗余能力，存在重大的资源或容量处理能力的损失，或主要系统组件的完全丢失。解决方案（这里的解决方案是指解决或者绕过问题的方法）可能存在，但它的使用并不理想。
“严重等级3” : 意味着一些造成程序不太重要部分错误的问题。比如监管功能（包括但不限于）丢失, 或者该错误对系统有很小的影响。 有一个比较合理满意的解决方案，但存在的问题可能会导致用户不满。
“严重等级4” : 是指一个问题，对系统有很小影响或者非关键方面。这些包括但不限于一般的文档问题或界面显示问题。 有合理满意的解决方案存在，不对系统的使用造成实质性的损害。
