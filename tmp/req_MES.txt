# 工厂基础建模
## Factory base data modeling （工厂基础数据建模）
- 可支持 FAB/区域/设备对应关系，设备可以分配到不同的区域用于工艺工程师/设备工程师/制造目的。支持产品设置（工艺--产品--生产流程--工序--工步--对应的 recipe 与机台关系的MES 架构模式），包括产品工艺信息的定义
## Equipment Modeling （设备建模）
- 设备分类管理与定义：需满足Litho Inline Tool/Chamber Tool/internal Buffer Tool/fix buffer tool/Stocker，同时equipment应可自定义和转换。
- 设备能力的定义：（offline，online local,online remote…)
- 设备配方及参数等配置设置
- 支持设备Unit的定义： chamber, Header, Tank 等的支持
- 设备批量数量的定义,例如：最大和最小Wafer数的定义，最大和最小载体数量的定义
## Stocker Modeling
- can define stocker : wafer stocker, reticle stocker 支持定义Stocker的类型：wafer STK, Reticle STK...
- Can define storage form of stocker.STK的容量的定义
## Process basic data modeling
- 产品:定义工厂生产的成品/半成品等；产品定义及其属性（可以定义产品的wafer类型，产品责任人）
- 设置产品的基本信息时，可为产品指定一个或多个流程，并指定缺省流程
- 支持定义产品组（产品的Type定义）、不同工艺的产品类别、Reticle的设置（能定义不同产品的Photo Layer 的光罩设置）
- 机台配方:设备配方的定义
- 版本的控制要能支持Active版本的定义, 每个Object 只有唯一的Active 版本号. 流程支持版本和状态（Unfrozen/Frozen/Active）管理，流程的版本在Unfrozen状态下可以修改，在Frozen和Active状态下无法修改，激活后的版本才会生效。版本Active后不能进行修改和删除，如果要修改，必须升级版本后，对新版本进行修改，然后对新版本进行Active。
## Code Define
- 提供Reason Code的定义：Reason的定义用于各种功能，如hold、rework...等。
- 提供机况定义及其对应的（E10）状态，设置不同机况下的跑货限制，机况转换规则等
- 提供用户自定义代码定义：增加属性扩展的灵活性，方便使用自定义逻辑。
## Lot Type Define
- 提供批次类型和批次子类型定义
- 提供基于批次子类型的批次id命名规则定义，如起始字符
- 系统可以为特定批次子类型运行时的机器状态转换提供不同的规则定义
## Lot naming
- LotID命名规则支持通过配置方式自定义，满足工厂多样性的要求
- 当进行批次分批时，应自动生成子批次LotID
## 用户组
- 用户可以创建一个用户组，该组具有相同的权限
- MES可以按组管理用户控制
- 每个用户都属于某个用户组，他们会被用户组管理或控制，一个命令作用于每个用户
## 权限管理
- 操作安全管理（用户/用户组添加/更新/删除）
- 对关键操作的双重确认支持
- 由用户/用户组控制的功能权限
- 用户部门控制的设备权限
- 由用户部门控制的持有/释放权限
- 由菜单和操作按钮都要可以控制权限，系统默认要对按钮具备权限管控不能只是对菜单权限管控
- 支持用户组的角色配置，为角色分配权限。可以在菜单、按钮级别明确权限
- 支持设备/区域和权限设置，操作者只能使用属于指定设备的设备
- 系统的菜单布局和顺序应该默认统一。 相同权限组 或者 用户组 的用户登陆系统后需要看到的菜单布局完全一致，方便提高工作效率。
- 不同权限组 或者 用户组 的用户登陆系统后，系统菜单布局 需要 按照默认统一的顺序展示对应具备权限的菜单，而不能无序展示菜单，让其他用户无法快速定位功能菜单位置， 严重影响工作效率
- 支持操作员设备操作证书管理，只有持有证书的操作员才能使用设备，支持对接OA系统
# 流程建模和管理
## Process Specification Editor
- Process Specification Editor是一个类似于Windows-Explorer风格的查看器，它提供了一个带有Process Planning信息的层次结构.(小批量修改的功能，不需要导数据,一些小的数据，直接在这个编辑器里改)
- 用户可以更改其值并在运行时应用它 (大批量修改是批量导入,小批量修改直接在编辑器里面进行修改即可）
## Recipe
- 支持多腔设备，支持腔体串并联
- 支持recipe生命周期管理
- 可对recipe的类型进行设置（制程/测量等），并可以定义recipe 所属区域。
- 不同的产品或产品组下可以挂不同的设备
## Flow
- 在流程和配方之间需要有一个多层模块化结构，至少有一个额外的层（模块或子计划），以便于未来的维护
- 模块可以在流程内部和流程之间重复使用。如果系统提供的流程结构相对简单，则有必要提供高效准确的系统解决方案来应对未来的大量变化
- 一个流程可以被多个产品使用。支持Bond/Debond
- 支持临时Bond 和 永久Bond
- 提供流程内部循环数设置，可在批量界面查看
- 提供构建流程模板，可以快速直观的构建流程。可以直接导入到系统中。在导入之前，应在规则上预先检查流程
- Active Flow 或者任何一个版本的 Flow 均支持导出
## Multipath Flow
- 允许手动或在预设条件下灵活跳转到正常产品流上的其他分支流程
- 预设条件包含：手动，byProd，ByEDC等
- byProd：针对不同Prod公用一个Process的场景，可以按照不同ProdID自动选择走不同的分支
- ByEDC：针对量测结果的不同区间，自动选择该区间对应的分支
## Reticle
- 需要 Reticle 组的概念。可以将相同类型的光刻分组为一组。实际在线使用时，可以选择其中之一，完成光刻
- 系统可以查询每个产品的整套Reticle信息
## Product
- 不同prod可以共用同一个Flow，Prod可以设定基本属性
- 具备ProdAttribute功能，可针对Recipe，Reticle，EDCPlan进行动态匹配
## Rework
- Rework次数到了，还要有超级权限可以继续做rework
- 支持By Prod， By Process设定 Rework Count
- Rework 需要具备 subplan 级别的计算方式 支持不同工步返工次数的绑定计算，共享相同的返工次数
## Version Control
- Process，Product，Subplan要具备 版本控制功能，都要具备历史查询详细的记录
- Flow 支持版本升级，默认使用最新版本
- 需要具备人性化的Flow版本比对功能
## Flip
- 可以设定Wafer的正反面 Flip属性
- 需在流程管控体系中支持Flip的各种防呆卡控
## ProcessLocation
- 支持不同类型载具在流程中不同step的防呆卡控，例如槽位数量不同需卡控，前后段制程转换需卡控
## DedicationFlag
- 支持不同Stage，Layer 卡控必须使用相同的机台或者机台组，前量和后量卡控同一台设备
## Loop Control
- Loop次数管理，次数没到上限就自动返回Loop起始站点继续加工，次数达到上限就自动跳出Loop循环
## EDCPlan
- 流程上的站点可以设定 EDCPlan
## 站点污染等级
- byStep设定污染等级，Lot在该站点出站后的污染等级变更为该站点设定的污染等级
## Q-time
- 支持Qtime嵌套，交叉等设定，支持TrackOut触发，TrackIn触发，TrackIn结束，TrackOut结束
- Lot支持多个Qtime
- Q-Time设置和Q-Time动作：提供最小和最大Q-Time，多步之间的Q-Time，并提供不同的触发设置，当系统超过Q-Time时自动触发，不同的时间触发不同的警告
- 具备CancelTrackIn 后Qtime恢复计时的能力，当Lot 处在Q-Time 结束站点时, 如果发生Cancel TrackIn, Q-Time需要继承回来.
- 分批时, 子批要能自动继承母批的Q-Time
- 如果 lot 在加工过程中 Over Q-Time, 要能够触发Track Out 后 Hold Lot动作
- 具备Wafer级别的Qtime记录能力，lot 分批 或者 合批 后 Qtime仍然有wafer级别的单独记录，可以为后续操作提供精准的Qtime数据
- 支持强制关闭Qtime的功能
- 支持设定 DummyStepForQtime，人工进行判定是否放行 Qtime
## BankFlag
- 可定义允许做Bank操作的站点
## 导入功能
- Flow，Subplan，Step等都要支持Excel模版导入，提高工作效率
# 机台建模和管理
## EQP基本属性
- 设备属性/能力/类型管理，设备名称、描述、组别、Batchsize、腔室、module等
## Define Equipment group
- Equipment group下面可以挂多个机台
## Define chamber
- 可以定义腔室ID。可以定义串行/并行/组合三种模式
- Chamber机台增加新的chamber时，只需要添加chamber机台（自动继承父设备属性），无需修改其他Flow/Recipe等属性
## Set Multi Chamber Recipe Mapping
- chamber RCP系统功能。针对并行模式，需要具备一键关闭某一个chamber就把所有带有该chamber的recipe全部都关闭的能力
## Equipment status define
- 提供用户定义的机器状态转换规则 以及 对应的权限
- 提供原始设备状态的定义，以转换为 MES 定义的设备状态
- 机台状态切换规则，以及不同 LotType 例如 MonitorLot 进出机台时，机台状态要切换到特定状态，不能简单的 Run-Idle 模式
## Chamber status define
- 腔室状态与批次进出站联动，腔室状态将对应转换为不同的设备状态。
- 主设备状态将根据腔室状态合理的自动更改。
- 相应的Recipe由设备和腔室状态组合定义。当执行该过程时，系统会选择最佳的可运行配方（RCP系统管控）。
- chamber 状态可以跟父设备联动，也可以单独切换
## ECS
- 具备PN/RN：正向列表：根据限制规则设置，允许 Lot 派工；负向列表：根据限制规则设置，禁止 Lot 派工
- 提供设置在特定条件下可以根据时间间隔自动重新启动RN
- 设置的禁止条件可以按照 IdleTime，LotCount(Total/Daily)，WaferCount(Total/Daily)
- 支持多种的条件组合定义：产品/Flow/LotID/LotType/StepID/Priority/stageID/Recipe/设备/腔室/Reticle等。
- 支持例外条件，在卡控条件基础上 可以 让 满足例外条件的 获得 通过，提高用户设定的工作效率
- 支持通配符设定卡控条件和例外条件
- 需具备Doublecheck才能生效的机制
## EQP 查询和History查询
- 支持by各种条件筛选查询
- 支持查询条件的文本框内直接输入模糊字符（例如带有*号）后点击查询按钮就可以进行查询，不可以是 先点击下拉菜单获取数据后再模糊匹配选择确定数据再查询的方式
- 查询结果显示要具备分页查询功能，不得将所有数据一起显示造成系统卡死或者闪退
- 需具备Chamber级别的 Wafer过帐记录的能力：ElogSheet功能
## 炉管基础功能
- 支持多种batch方式：生产批次+炉管Monitor lot
- 仅生产批次
- 仅Monitor批次
- 生产批次+RCLot+炉管MonitorLot
- RCLot+MonitorLot
- 生产批次+RCLot
- 生产批次+RCLot+炉管MonitorLot+非炉管区的前制程需长膜的MonitorLot
- 通过ARG系统针对BatchLot的Wafercount区间，进行layout recipe动态调整
- 组Batch时需针对batch内多个lot的 Recipe，EDC等信息进行合理的卡控
- 炉管MonitorLot做完量测后，可以对Batch内的生产批次进行同步的量测过帐以及量测数据贴值
- 炉管Batch Run异常后，可以先DeleteBatch，然后MonitorLot 和 Batch内其他Lot需要各自走自己Flow
- 炉管的Fill/Side DummyLot 可以EAPAuto进出站
## WET基础功能
- 可以选择相同recipe的Lot在WET进行 单批 或者 2个批次 batchRun
## Sorter设备
- 支持SorterJob基本类型，如：Exchange，Split，Merge，WaferFlip，ReadT7Code
- Sorter 机台要能支持流程内 (Inline Sorter) 和非流程 (Adhoc Sorter)的作业
- 支持Inline Sorter step, 能够按照用户的设定完成 Sorter动作：例如Split，Flip，ReadT7Code等
- 支持SorterJob 的查询和取消
## 机台可接受污染等级
- 机台设定可接受的Lot的污染等级，支持设定多个污染等级。 污染等级需要精确匹配
# 载具管理
## Create Carrier
- 允许用户创建 Carrier。用户可以根据 Carrier 类型类别和Carrier 型号创建指定数量的 Carrier
## Assign Carrier
- 允许用户将 carrier 分配给没有 carrier 的批次。指定的载体必须是Free的，晶片的顺序可以在该功能中更改。
## Exchange Carrier
- 允许用户交换载具。原始 Carrier 和交换 Carrier 应为同一类型
## Hold Carrier
- 允许用户HoldCST，CST状态应更改为 HOLD。
## Release Carrier
- 允许用户放行CST。完成此功能后，CST应处于Hold之前的状态。
## Clean Carrier
- 允许用户为任何 Carrier 进行清理操作。设置完成后，其状态应为WaitClean。完成清洗后，状态应为Free
## 载具查询功能和历史查询
- 允许用户查询 Carrier 的状态，历史，实时信息，装载的LotID等。
- 可以根据用户选择的不同条件查询进行查询
- 支持查询条件的文本框内直接输入模糊字符（例如带有*号）后点击查询按钮就可以进行查询，不可以是 先点击下拉菜单获取数据后再模糊匹配选择确定数据再查询的方式
- 查询结果显示要具备分页查询功能，不得将所有数据一起显示造成系统卡死或者闪退
## Change Slot map Info
- 允许用户更改同一 Carrier 中晶片和原始批次之间的关系
## Modify Carrier Attribute
- 允许用户修改 Carrier 的属性
## 载具类型卡控
- CST 跟 Processlocation的mapping关系需要可以配置，实现流程管控中的CST类型防呆卡控
## PCD
- Carrier 需要由 cassette/pod/door 作为属性组成，RFID内写入的是 carrier id（RFID 安装在 Cassette 中）， cassette/pod/door 扫码输入，系统以 carrier id 做索引
- 支持清洗周期管理
- 支持清洗机台的PCD过帐管理
# Reticle管理
## Add Reticle
- 工程师可以通过特定的命名规则添加新的reticle ID, pod, film type, max wafer clean count, max lot clean count等等
## Hold Reticle
- 工程师可以使用此功能来 Hold Reticle。在 Hold Reticle 功能中，用户需要输入 Hold Code 和注释
## Release Reticle
- 工程师可以使用此功能 Release Reticle。在 Release Reticle功能中，用户需要输入 release code 和注释
## Return Reticle
- 如果 Reticle 不再使用，工程师可以将 Reticle ID 返回到 Reticle版室
## Scrap Reticle
- 工程师可以使用此功能来报废 Reticle。在报废 Reticle 功能中，用户需要输入Scrap code和注释
## UnScrap Reticle
- 如果 Reticle 已经报废，经过工程师的验证，Reticle 仍然可以使用，工程师可以使用此功能来解开 Reticle
## Bank In Reticle
- 工程师可以使用此功能将需要的Reticle移动到 mask 室的。
## Bank Out Reticle
- 如果经过工程师的验证，Reticle 仍然可以使用，工程师可以使用此功能将 Reticle 从 mask 室中取出
## Modify Reticle
- 工程师可以修改Reticle的各种属性
## Inspect Reticle
- 工程师可以使用此功能来检查 Reticle
## Clean Reticle
- 如果需要清洁 Reticle，工程师可以清洁 Reticle ID
## Reticle Location
- 支持记录Reticle位置，例如Stocker，机台，Maskroom等
## Reticle Pod
- 工程师可以为 Reticle 添加一个 Pod ID。Pod 信息包括Pod ID、Pod 类型、Pod 容量、清洁周期等
## Change Reticle Status
- 可以通过配置方式实现各种状态和切换规则
## Reticle TrackIn/TrackOut
- 支持在特定Litho机台做Retcile的进出站过帐并记录对应历史
- Lot所需的Reticle进站后，lot才允许进站
- Lot使用的Reticle需要等待Lot出站后，才允许出站
- 支持机台最大TrackIn的Reticle数量卡控，超出数量不允许TrackIn
## Printdown
- 设置PDFlag
- 设置PD最大使用次数 和 有效期
- Max Using Count超限后，卡控批次不能进站，状态切为WAIT_PD，Track In后Status切为IN_PD
- The system keeps a history of reticle and reticle pod state change record
## IRIS
- 设置IRISFlag
- 设置IRIS的管控Spec
- 设置IRIS最大曝光wafer数量
- 支持IRIS收值和判定
## 查询主界面和历史查询
- 工程师可以使用此功能查看 Reticle 的详细信息，用户可以根据 Reticle ID、group, pod and EQP 来选择 Reticle 的具体信息
- 可以根据用户选择的不同条件查询进行查询
- 支持查询条件的文本框内直接输入模糊字符（例如带有*号）后点击查询按钮就可以进行查询，不可以是 先点击下拉菜单获取数据后再模糊匹配选择确定数据再查询的方式
- 查询结果显示要具备分页查询功能，不得将所有数据一起显示造成系统卡死或者闪退
## Reticle Group
- 支持设定ReticleGroup，下面关联多个Reticle
# Lot管理
## Lot plan
- 可以跟ERP系统联动，创建新的Lot
- 可以跟ERP系统联动，创建半成品Lot
- 可以支持在MES系统中独立创建Lot
- 支持Batch创建Lot
- 支持创建Lot的wafer选择不同物料，而不是 同一个Lot的wafer只能用一种物料
- 支持创建Lot时具备记录PO，Order，客户信息等 功能
- 创建Lot的Qty可以自定义
- 创建Lot ID时 可以根据LotType等信息自动进行编码
- 未到规定的下线时间不允许下线Lot
- 支持StartHold功能，并且可选开启还是关闭
- 物料不足时，不允许创建Lot
## Cancel lot plan
- 创建的Lot还未下线之前，可以取消创建Lot，并且LotID编码不会被占用
## Change lot plan
- 可以修改Plan好的Lot的Lot类型
- 可以修改Plan好的Lot的优先级
- 可以修改Plan好的Lot的StartHold
- 可以修改Plan好的Lot的Owner
- 可以修改Plan好的Lot的CustomerID
- 可以修改Plan好的Lot的Plan Start Day/Due Day
- 所有修改需要具备历史记录
## Lot Plan Portal 和 历史查询
- 工程师可以使用此功能查看 Lot的详细信息，可自定义结果排序
- 可以根据用户选择的不同条件查询进行查询，Lot ID条件 支持同时输入逗号分隔的多个LotID模糊查询
- 支持查询条件的文本框内直接输入模糊字符（例如带有*号）后点击查询按钮就可以进行查询，不可以是 先点击下拉菜单获取数据后再模糊匹配选择确定数据再查询的方式
- 查询结果显示要具备分页查询功能，不得将所有数据一起显示造成系统卡死或者闪退
## Bank In/Out
- 可以单个 lot 或多个 lot 同时做 BankIn/BankOut 动作.
## Ship/UnShip
- 支持 Lot Ship以及取消Ship
## Scrap/UnScrap
- 支持 Lot 整批报废以及取消报废
## Terminate/UnTerminate

- 指定物料，byWafer指定物料
## 分批/合批
- 需要支持逻辑/物理分批
- 物理分批动作后，要自动触发分批的 Sorter Job，后续使用 Sorter 站点标准作业进行分批
- 需要支持逻辑/物理合批
- 物理合批动作后，要自动触发并批的 Sorter Job，后续使用 Sorter 站点标准作业进行合批
- 当分批时,子批默认继承母批的属性,包括 FutureAction，Qtime, wafer 的前量数据，加工站所有信息等
- 合批时需要具备防呆卡控，确保Lot核心属性一致，不得产生MO风险，例如 Lot Flow版本，站点，污染等级，优先级等等
- 当子批 Merge 到母批时,子批的 Future Action 等信息需要 Merge 到母批上
## Hold/Release
- 可以同时用不同原因 (Reason Code) 对同一个Lot 加以Hold
- 支持 Batch Hold
- Hold 和 Release 要有历史记录, 可以清晰的跟踪 release 动作对应的是那个Hold动作, 历史要可以查询
- Release hold 时须选择相应的 hold code，和填写 release comment
- 根据 Department 以及Reason Code 管控Hold/Release 动作时的权限
- 单一 Lot 有多个 Hold Code, 可以选择多个 hold 使用相同的 Release Code 一次 Release
- 具备强制将机台上发生异常的Lot 做 出站 操作的系统功能
- Running Hold：因机台故障等各种原因而将所跑的 Lot Hold 住，再使用 Recovery Run Card 进行后续处理
## FutureHold
- 支持 Lot 在现在的 Flow 上，对未来某个站点设置 Future Hold
- 按生效时间点不同, FutureHold 可以分为 PreFutureHold (进站就Hold)和 PostFutureHold (TrackOut 时Hold).
- Lot 的 Future Hold 支持 Multi Future Hold 的设置. Future Hold 的设置可以被更改, 删除
- Lot Future Hold生效后,与版本无关 (版本升级后Future Hold应该被继承)

## Reassign

- 支持By ProductChange Reassign。按Product查询和选择一个或多个Lot，指定目标产品（使用产品当前激活的流程和版本），进行Reassign。系统自动Reassign每个Lot到目标产品和流程上，并定位到与Lot当前Step Seq相同的Step。如果目标产品流程上没有Lot对应的Step，Reassign失败并提示。可选择是否保留FutureHold。

# Bond/Debond
## Bond/Debond设定
- ByProd设定 主Prod和副Prod的绑定关系
- 支持WaferBond时的 TopProd/BottemProd关系的设置，并且需要跟EAPAuto联动
- 支持主副Bond站点的联动设定
- 支持主副Debond站点的联动设定
- 支持副Prod的Loop循环使用设定
- Bond站点需要设定特殊标记，不得被混用
- Debond站点需要设定特殊标记，不得被混用
- 支持永久Bond
- 支持临时Bond
## Bond/Debond操作
- 主次产品批次一起作业，Dispatch前或Move Out时指定Bonding关系。在指定Bonding关系时，可调整主次产品批次的Wafer顺序，系统成对绑定
- 如果是Dispatch前指定Bonding关系，则如果没有指定Bonding关系，Lot不能进站加工。
- Bond后，此产品的Lot状态变为Bonded，系统只操作主产品Lot
- 支持多层Bond，既可以Bond后，再Bond
- 支持Debonding，走过带有Debonding逻辑的工步，自动Debonding。Debond出来的批次，需要指定合法的新Carrier。
- Debond也支持多层Debond，可以指定Debond哪一层，缺省是后Bond的先Debond
- 次产品可以定义完整的Flow，如果可以Debonding，可设定循环次数，并进行控制
- 可以查询主次产品批次的完整Wafer历史
- 可以追溯Wafer到Wafer的Mapping关系
# Run Card management
## Run card Type
- 支持Split Run Card（SRC）和Recover Run Card（RRC）。SRC一般由PIE根据实验要求开具，RRC则一般由MA在发生异常后开具
## RC常规功能
- 每个RC包括：一个Split Step、多个Loop Flow、一个Merge Step；在Split Step，可以不进行分批，整个母批进入RC
- RC可以连续执行多个，中间无需走主流程Step
- 支持自动Split和Merge，或人工Split和Merge
- RC需要审核和激活，可按照RC的开具的部门，走不同的审核流程，签核顺序需要可调整
- Loop Flow，可以来源或基于现有Flow进行修改；长度没有限制，可以重用；可以指定EQP/EQP Group、指定Recipe、Check RCP、指定量测选片等
- RC Lot与普通Lot共用Dispatch界面、各种Trans界面、查询界面，以及Move In和Move Out逻辑…
- Loop Flow每个Step可以设定：Recipe、EQP，EQPGroup、Reticle、EDC Plan、EDC Wafer List、EDCSpec等
- RC Lot与普通Lot共用界面、各种Trans界面、查询界面，以及Move In和Move Out逻辑…
- 支持RC EDC数据采集进SPC，并且支持手动勾选开关是否进SPC
## SRC
- 支持当站开SRC或者在未来步骤开 FutureSRC，已经有FH（包括byProdFH）的站点需要可以开RC
- 支持SRC发生异常嵌套生成RRC的功能
- SRC支持跟APC等系统集成
- SRC支持RCP系统联动，并且有对应的设置防呆卡控
- 支持跳步RC等特殊RC功能
## RRC
- 支持PartialRRC
- 支持RRC 方式Rework
- 支持RRC方式嵌套SRC
- 支持Bond/Debond场景的RRC
## RC Qtime
- 支持设定 RC Qtime
- 支持在RC设定时自动识别主FlowQtime 结束站点，并且在RC内继承该Qtime 结束站点的任务
- 支持Lot在主Flow 开始一个Qtime，且在RC内结束该Qtime
- 支持在RC设定时自动识别主FlowQtime 起始站点，并且在RC内继承该Qtime 起始站点的任务
- 支持Lot在RC内触发继承自主Flow的Qtime计时，RC结束回到主Flow后 结束该Qtime
- 支持在RC内单独设定Qtime开始和结束站点
- RC内单独设定的Qtime开始和结束站点必须配对出现，不得缺失设定导致Qtime无法正常结束

- 支持管理和调整挡控片的已经使用次数（By Wafer Level）

- 支持手动上传挡控片测机数据
- 支持量测设备单独做AM的方案
- 要支持Multilot模式测机

- 对于炉管Monitor片，在组Batch时，需要指定Monitor Lot。在进行炉管加工后，Monitor片会进行测量，测量后的值，会自动复制到Batch内的Product Lot。
- 一个炉管Batch内可包含多个Product的Lot，但这些Lot的加工Recipe必须相同。如果进行量测，这些Lot的抽片方式不同，则必须在量测前，打散并重组Batch，保证每个量测的Batch内的Lot都有相同的抽片方式。
- 支持A/B Batch
## Season
- 能够定义 season route, season lot, season recipe. 系统应该支持按用户配置的信息做 Season Split
- 按照ByPM触发Season，且支持多个SeasonProdLot做 ByPM的season
- 按时间 (Eqp/Recipe idle时间) 触发Season
- 按Wafer数量 (持续/累计) 触发Season
- 按recipe group变化触发Season
- 提供 GUI 给用户维护 Season 信息,如:Query,Add,Modify,View,History,Copy 等
- Season支持自动逻辑分批的SeasonLot进行season，无需人工参与
- byPM的 By Step Season Complete 和 cancel，中途down机的场景支持
- CMP zero-idle: season lot 和 production lot 需要确保连续RUN
- Chamber 的 season 要以 chamber 实际 Idle 的时间为准, 而非以主机台时间为准
- 当 season 出现某些异常, 用户可以手动完成 season 或重置season 状态及信息,以使得重新做season.
- 暖机控制对象包括主机台和子机台, MES在全自动, 半自动及手动模式下都需要支持Season控制
- 支持多次暖机，即机台需要用多种Season wafer跑多个程序才完成暖机要求，且有时效管控

## 工单管理

- 11) 根据工单和批次可以方便的查看到所用的工艺流程和生产BOM等；
- 12)支持按工单信息查看，包括已完成数量，生产中数量，报废数量等信息
- 13)提供追溯查询，查看批次的作业加工履历、物料耗用数据、工时数据等；
- 14) 工单完工结算：从物料耗用、工时核算两个大的角度对工单进行结算，并将结算的数据回报给ERP。
